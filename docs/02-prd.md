# 02 · 기획서 (PRD)

> Markdown Radio — AI 정보 소비용 듣기↔읽기 폐루프 웹앱
> 버전 0.2 · 2026-06-21

---

## 1. 한 줄 정의

**마크다운 자료를 청취용으로 의미 정제해 읽어주고, 흘려듣다 북마크한 지점을 원문의 정확한 문장으로 되돌려주는 듣기↔읽기 폐루프 웹앱. 무료·온디바이스 청취를 최종 지향으로 한다.**

> 즉 핵심은 두 가지 해자 — (1) 마크다운을 기계적으로가 아니라 **의미를 이해해 청취용으로 정제**하는 것, (2) **북마크 → 원문 정확 점프**로 흘려듣기(1단계)와 정독(2단계)을 잇는 **폐루프**다. 무료·완전 온디바이스(Supertonic)는 이 제품의 정체성이자 최종 지향점이며, 초기 가치검증 단계에서는 브라우저 내장 Web Speech를 부트스트랩 경로로 쓸 수 있다(§7 전략 노트 참조).

---

## 2. 배경과 문제 정의

### 2.1 문제

AI 검색(Perplexity, Gemini 등), 논문, 테크 블로그에서 쏟아지는 **마크다운 기반 자료의 양이 과다**하다. 모든 자료를 눈으로 정독하기엔:
- 피로감이 크고 시간이 오래 걸린다.
- 어떤 자료가 정독할 가치가 있는지 **사전 필터링**이 어렵다.
- 이동 중(출퇴근·산책)에는 화면을 볼 수 없어 자료 소비가 멈춘다.

### 2.2 해결 가설

정보 습득을 **두 단계로 물리적으로 분리**한다:

| 단계 | 행동 | 목적 | 상황 |
|------|------|------|------|
| **1단계** | 귀로 흘려듣기 | 맥락 파악 · 정독할 자료 필터링 | 이동 중, 멀티태스킹 |
| **2단계** | 눈으로 정독 | 중요 내용 깊이 읽기 | 책상 앞, 집중 시간 |

두 단계를 잇는 다리가 **북마크**다. 1단계에서 "어, 이거 중요한데?" 싶을 때 버튼을 누르면 그 지점이 기록되고, 2단계에서 북마크만 열어 해당 문맥을 바로 정독한다.

### 2.3 왜 지금, 왜 이렇게

- 2026년 기준 **브라우저 안에서 100% 로컬로 도는 한국어 TTS 모델**(Supertonic)이 실용 수준에 도달 → 서버·비용 없이 무료로 가능.
- 기존 TTS 앱/서비스는 (a)클라우드 비용·프라이버시 문제, (b)마크다운 자료 특화 부재, (c)"흘려듣기→북마크→정독" 워크플로 부재.

## 3. 타겟 사용자

### 3.1 주 사용자 (페르소나)

**"정보 과식자" — 개발자/연구자/지식 노동자**
- AI 검색·논문·기술 문서를 하루에도 수십 건 마주친다.
- 출퇴근/산책 시간을 정보 소비에 쓰고 싶어 한다 (에어팟 상시 착용).
- 한국어 자료가 중심이지만 코드·영문 용어가 섞여 있다.
- 프라이버시·비용에 민감해 "내 자료가 서버로 안 가는" 온디바이스를 선호한다.

### 3.2 사용 환경

- **PC**: 책상에서 자료 정리하며 청취 + 정독 — **폐루프 검증의 주 무대**(흘려듣기↔정독이 한 자리에서 일어남).
- **모바일(Android)**: 이동 중 청취. 백그라운드 재생에 유리.
- **모바일(iOS)**: 이동 중 청취. 백그라운드 재생에 제약이 있어(§9) **후순위**로 둔다. 첫 스프린트에서는 제약을 명시한 채 제한적으로만 지원한다.

## 4. 핵심 가치 제안

순서는 차별화 우선순위(해자 → 정체성·방향성 → 사용 맥락)를 따른다.

1. **마크다운을 의미로 이해해 정제한다.** [핵심 해자] `#`, `*`, 코드블록, 표, 링크를 기계적으로 읽지 않고 **무엇을 소리 내고 무엇을 버릴지**를 의미 단위로 판단해 청취에 자연스러운 흐름으로 다듬는다.
2. **북마크 → 원문 정확 점프로 폐루프를 닫는다.** [핵심 해자] 흘려듣다(1단계) 누른 북마크가 정독 뷰(2단계)에서 **원문의 정확한 그 문장**으로 되돌려준다. 듣기와 읽기가 끊기지 않고 이어진다.
3. **무료·온디바이스를 최종 지향한다.** [정체성·방향성] 사용자 기기가 연산하므로 서버·API 비용이 구조적으로 0이고, 자료가 기기를 떠나지 않아 완전한 프라이버시·오프라인 동작이 가능하다 — 이것이 제품의 최종 형태다. (v0 가치검증 단계에서는 브라우저 내장 Web Speech를 부트스트랩으로 쓸 수 있다 — §7 전략 노트.)
4. **이동 중에도 정보 소비가 멈추지 않는다.** 화면을 볼 수 없는 출퇴근·산책 시간을 귀로 채운다.

## 4-A. 경쟁 포지셔닝

읽어주는 기능 자체는 이미 흔하다. 차별점은 **마크다운 의미 정제 + 북마크→원문 점프 폐루프**의 결합이다.

| 항목 | Edge 소리내어 읽기 | Pocket | Readwise (Reader) | Speechify | **Markdown Radio** |
|------|------------------|--------|-------------------|-----------|--------------------|
| 마크다운 의미 정제 | ✕ (웹페이지 본문 낭독) | ✕ | △ (하이라이트 중심) | ✕ | **◎ 의미 단위 정제** |
| 북마크 → 원문 정확 점프(폐루프) | ✕ | △ (저장만) | △ (하이라이트 보관) | ✕ | **◎ ±1문장 점프** |
| 무료 | ◎ (브라우저 내장) | △ | ✕ (유료) | ✕ (유료) | **◎ 구조적 0원** |
| 온디바이스/프라이버시 | ✕ (클라우드 음성) | ✕ | ✕ | ✕ | **◎ 완전 로컬(최종 지향)** |
| 음질 | △~○ | — | — | ◎ | ○ (Supertonic 목표) |

**결론**: 우리만의 자리는 **폐루프(마크다운 의미 정제 + 북마크→원문 점프)** 다. 음질·무료·온디바이스는 이 자리를 지지하는 **보조 우위**이지 단독 차별점이 아니다 — Speechify는 음질로, Edge는 무료로 이미 강하다. 따라서 검증·메시징의 1순위는 항상 폐루프다.

## 5. 핵심 사용자 시나리오

### 시나리오 A — 출근길 흘려듣기
1. 전날 밤 PC에서 AI 검색 결과 5건을 `.md`로 저장 → Markdown Radio에 업로드해 "재생 목록"에 넣어둔다.
2. 아침 지하철에서 폰으로 홈 화면 아이콘을 켜고, 에어팟으로 재생. 1.25배속.
3. 3번째 자료에서 흥미로운 대목 → **북마크 버튼 탭**. 계속 듣는다.
4. 사무실 도착. 북마크 2개가 기록돼 있다.

### 시나리오 B — 책상에서 정독
1. PC에서 북마크 목록을 연다.
2. 북마크를 클릭 → 원문에서 해당 문장으로 **자동 스크롤 + 하이라이트**.
3. 앞뒤 문맥을 눈으로 정독. 필요하면 그 지점부터 다시 재생.

### 시나리오 C — 오프라인
1. 비행기 모드. 이미 1회 다운로드된 TTS 모델이 IndexedDB에 있으므로 그대로 동작.
2. 저장해 둔 문서를 청취. (새 모델 다운로드만 불가)

## 7. 범위 (Scope)

범위 설계의 제1원칙은 **폐루프(흘려듣기 → 북마크 → 정독 점프)를 가장 먼저 검증**하는 것이다. 음질 엔진을 완성하기 전에 "흘려듣다 북마크하고 그게 실제로 정독으로 이어지는가"라는 가설을 먼저 깨야 한다.

### 7.1 MVP (1차 스프린트) — In Scope · 폐루프 검증 우선

핵심은 **마크다운 의미 정제 + 흘려듣기 + 북마크 + 정독 점프**의 한 바퀴를 PC에서 끝까지 도는 것.

- 문서 입력: 파일 업로드(`.md`, `.txt`) + 텍스트 직접 붙여넣기
- 마크다운 **의미 정제** (헤더/강조/링크/코드블록/표/리스트 처리) — 핵심 해자
- 문장 청크 분할 + 원문 위치 메타
- 연속 재생 (재생/일시정지/이전·다음 문장/seek) + 배속(피치 보존)
- 북마크 (추가/삭제/목록) + 정독 뷰(원문 하이라이트·**±1문장 점프**) — 핵심 해자
- 문서 라이브러리 (저장/삭제, IndexedDB)
- 동작 무대: **PC(Chrome/Edge)** 우선

> **TTS 엔진 전략 노트 (codex 제동 준수).** 제품의 정체성·최종 지향은 **온디바이스 Supertonic** 그대로다(§6 결정 불변). 다만 폐루프 가치검증이 음질 엔진 완성에 묶이면 안 되므로, **1차 스프린트의 가치검증은 브라우저 내장 Web Speech API를 부트스트랩 음성 경로로 써도 된다**. Web Speech는 어디까지나 *가설 검증을 빨리 돌리기 위한 임시 경로*이며 제품 정의·메시징에는 등장하지 않는다. Supertonic 실청취·엔진 PoC는 이와 **병행하되 폐루프 검증을 막지 않는다**(§11 M0).

### 7.2 Post-MVP (다음 단계) — 향후

폐루프 가설이 서고 나면 정체성(무료·온디바이스)과 모바일을 채운다.

- **Supertonic 온디바이스 합성** (transformers.js) — 부트스트랩 음성을 정체성 음성으로 교체
- **모델 IndexedDB 캐싱·진행률** (1회 다운로드 후 오프라인)
- **PWA** (홈 화면 추가, 오프라인, MediaSession 잠금화면 컨트롤)
- **Android 백그라운드 재생**
- **iOS 지원** — 백그라운드 제약을 명시한 채(§9), 후순위로 단계적 도입
- 모델 선택 UI (Supertonic / MeloTTS / 화자·속도 프리셋)
- 제목 전환 시 의도적 무음(라디오 코너 연출)
- 재생 목록(여러 문서 연속 재생) · 자동 진행
- URL 가져오기(웹페이지 → 본문 추출 → 읽기)
- 클라우드 동기화(선택적, 북마크/문서) — 단 온디바이스 원칙 유지
- 다국어(영어 자료 별도 모델 분기)
- 북마크에 메모/하이라이트 색상

### 7.3 Out of Scope (명시적 제외)

- 서버사이드 TTS·계정·로그인 (온디바이스 원칙 위배)
- 음성 클로닝·커스텀 보이스 학습
- 실시간 협업·공유 서버
- 네이티브 앱(iOS/Android 스토어) — 웹/PWA로 한정

## 10. 성공 지표 (개인 프로젝트 기준)

> **최우선 품질 기준 = 북마크 → 원문 ±1문장 매핑 정확도.** 폐루프가 제품의 자리이므로, 점프가 엉뚱한 문장으로 가면 제품 가치가 무너진다. 다른 어떤 지표보다 먼저 만족해야 한다.

### 10.1 폐루프 성공 게이트 — "정독 전환율"

핵심 가설은 *"흘려듣다 북마크한 것이 실제 정독으로 이어진다"* 이다. 이를 **정량 AND 정성**으로 함께 본다(어느 한쪽만으로 판정 금지). 지표 정의의 SSOT는 [03-functional-spec FN-13 · 계측](03-functional-spec.md)에 있다.

- **정량 게이트**: 24h 북마크 복귀율(=북마크 중 24시간 내 정독 뷰에서 다시 열린 비율) **또는** 청취 완료율 **≥ 40%**.
- **정성 게이트(2종 모두)**: ① 2주간 자발적 재방문 일수(도그푸딩 중 "다시 켜고 싶어 켠" 날), ② 북마크 재청취가 실제로 발생하는가.
- **표본 하한**: 북마크 **≥ 20개**가 쌓이기 전에는 정량값으로 판정하지 않는다(개인 n=1이라 표본이 적으면 노이즈).

> ⚠️ **이 게이트는 런칭 전 하드게이트가 아니라 런칭 후 실험 게이트다(codex 제동).** 개인 n=1 데이터로 단독 Go/No-Go를 내리지 않는다 — 방향을 잡는 신호로 쓰고, 폐루프 메시징·기능을 조정하는 근거로 삼는다.

### 10.2 보조 지표

- **정성**: "출퇴근길에 실제로 매일 쓰게 되는가?" (본인 도그푸딩)
- 첫 문장 재생까지 시간(2회차, 캐시 상태) < 3초
- 1시간 청취 중 체감 끊김 0회 (PC/Android)

## 11. 마일스톤

마일스톤은 **폐루프 가치검증을 본선(M1~M4)** 으로 놓고, **Supertonic 엔진 PoC는 비차단 병행 트랙(M0)** 으로 분리한다. 음질 PoC가 늦어도 폐루프 검증은 Web Speech 부트스트랩(§7 전략 노트)으로 계속 굴러간다.

| 단계 | 산출물 | 검증 |
|------|--------|------|
| **M0** 엔진 PoC *(병행·비차단)* | Supertonic 데모 **실청취** + 한 문장 합성 + 재생위치↔원문 **offset 검증** | 한국어 음질 체감 + 한 문장이 소리남 + 북마크 offset이 ±1문장 안 |
| **M1** 텍스트 파이프라인 | 업로드→**의미 정제**→청크→원문메타 | 샘플 MD가 깔끔한 평문 청크로 |
| **M2** 흘려듣기 재생 | (부트스트랩 음성으로) 연속 재생 + 배속 | 문서가 끊김 없이 흘러나옴 |
| **M3** 북마크·정독 폐루프 ★ | 북마크 저장/목록 + 정독 뷰 **±1문장 점프** | 흘려듣다 북마크 → 원문 정확 점프 (핵심 가설 검증) |
| **M4** 폐루프 실사용 검증 | 도그푸딩 + §10 게이트 측정 | 정독 전환율 정량·정성 신호 수집(런칭 후 실험 게이트) |
| **M5** 정체성 채우기 *(후순위)* | Supertonic 합성으로 교체 + 모델 IndexedDB 캐싱 | 온디바이스 음성으로 동일 폐루프 동작·오프라인 |
| **M6** PWA·모바일 *(후순위)* | SW+manifest+MediaSession + Android, iOS(제약 명시) | 홈 화면 추가, 잠금화면 컨트롤 |
| **M7** 배포 | Cloudflare Pages 배포 | 폰에서 URL 접속·설치 |

상세 태스크는 [04-checklist.md](04-checklist.md).
